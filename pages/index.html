Example Parade

Each of the visualizations linked below have different MDP simulator and optimization parameters but they are otherwise the same visualization implementation. When you change the parameters within the visualization the address of the page will update to save your current parameters. You can share or save parameter sets by copying and pasting the current address of the page.

Introduction: Explore with Help Messages

In this version of the visualization we enable help messages to illustrate the components. Since no parameters are specified, the MDP simulator and optimizer determine the initial parameters when it returns the initialization object. All the subsequent examples were constructed by changing the parameters from these initialized values.

Recommended interaction: Read the help messages and try performing each of the available interactions: (1) changing the parameters and optimizing the policy, (2) brushing the histograms, (3) changing the filtered year in the fan charts, (4) brushing the fan charts, (5) clicking one of the lines in the fan charts to display the associated rollout's state snapshots, (6) clicking a second set of rollouts in the history panel to view its comparison.

  • Help messages are on.
  • The server's default initialization is used.
  • The fire spread model has a lower fidelity to reduce execution time.
  • The bugs are on.

Check for Reward Sensitivity

We would like policies that are optimized to change as the rewards are changed. This activity explores whether the policies are "interesting."

Recommended interaction: Look at the coefficients of the current policy and/or the charts for the policy probabilities. Next change all the reward parameters to 0 and click the "optimize" button. A new policy will be optimized and new rollouts will return. Look at the updated coefficients and/or the charts for the policy probabilities. If the policy is sensitive to changing rewards, the coefficients should change sign or relative magnitude, and the distribution of policy action should shift. For a more direct comparison you can press the comparison button for the original policy's rollouts.

  • Help messages are off.
  • The policy was optimized for the default rewards.
  • The bugs are on.

Check for Concept Drift: Bug Fixes Applied

Following our use-case study with MDPvis, we patched numerous issues with the model. Here we set a flag, "Use Original Bugs," to 1 which opens a the fixed version of the MDP simulator and optimization algorithm. We would like to know whether these fixes created a simulator with significantly different behavior or optimized policies.

Recommended interaction: Turn the bugs on by changing "Use Original Bugs" to 0, then generate rollouts and compare the rollouts generated under the buggy and patched versions of the system.

  • The server's default initialization is used.
  • Bugs are turned off.

Validate Low Fidelity (Fast) Model

All the prior examples use a version of the model with less spatial resolution to speed exploration. We would like to know whether the inferences from exploring the domain are valid at higher resolutions.

Recommended interaction: Validate different model fidelities by changing the "Landscape Size" parameter and generating rollouts. Compare each of the resulting rollouts to each other. Ideally the percentiles of each fidelity will be equal and show a straight line across the fan charts.

  • The bugs are off.
  • 10 rollouts will be generated for 10 years each.
  • The initial policy is uniformly random.
About

MDPvis is a generalized visualization for testing and debugging Markov Decision Process domains and optimization implementations. The visualization is built to ease integration with problem domains and allow for exploring rollouts and rollout comparisons across many domain parameterizations.
